 The Code Again
func counter() func() int {
    count := 0

    return func() int {
        count++
        return count
    }
}
🔹 Step 1: What Normally Happens (No Closure)
Normally:

Local variables live on the stack

When a function returns, its stack frame is destroyed

Example:

func foo() int {
    x := 10
    return x
}
➡️ x is gone after foo() returns.

🔹 Step 2: Why count Cannot Live on the Stack
In counter():

You return a function

That function uses count

The returned function will be called later

So if count stayed on the stack:
❌ It would point to invalid memory

➡️ Go must keep count alive

🔹 Step 3: Escape Analysis (Compiler Decision)
Go compiler runs escape analysis and decides:

“count is used by a function that outlives counter()
Therefore, count must escape the stack.”

➡️ count is moved to the heap

This decision is made at compile time, not runtime.

You can verify:

go build -gcflags="-m"
You’ll see something like:

count escapes to heap
🔹 Step 4: What the Closure Actually Stores
The returned function is not just code.

Internally, a closure is roughly:

struct {
    fn_pointer
    &count
}
So:

count lives on the heap

The closure stores a reference (pointer) to it

Each call modifies the same memory

🔹 Step 5: Why count Is Still Alive
c := counter()
Memory situation:

Heap:
+-------+
| count | ← shared state
+-------+
     ↑
Closure function holds reference
Even though:

counter() returned

Its stack frame is gone

➡️ count survives because the closure references it

🔹 Step 6: Proof by Behavior
fmt.Println(c()) // 1
fmt.Println(c()) // 2
fmt.Println(c()) // 3
If count were recreated each time:
❌ Output would always be 1

But it increments:
✅ Same memory → same state

🔹 Step 7: Each Closure Has Its Own Heap Allocation
c1 := counter()
c2 := counter()

fmt.Println(c1()) // 1
fmt.Println(c2()) // 1
Why?

Each call to counter() creates:

A new heap allocation

A new closure referencing it

🔹 Important Clarification (Interview Gold)
❌ Go does NOT “always put closures on the heap”

✅ Go:

Uses stack by default

Moves variables to heap only if they escape

This is why Go is fast and memory-efficient.

🔹 One-Line Interview Explanation
“In this example, count escapes the stack because it’s captured by a returned closure, so Go allocates it on the heap and the closure holds a reference to it, allowing the state to persist after the outer function returns.”
